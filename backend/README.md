# VPNGate Manager backend

FastAPI, SQLAlchemy and Alembic backend for the VPNGate Multi-Exit Manager. Public VPNGate rows pass a strict OpenVPN whitelist before persistence and again immediately before privileged execution. Validated OpenVPN bytes are atomically copied to a root-owned `0600` runtime file before launch so the checked file cannot be swapped. Networking is exposed only as validated fixed operations through a root-owned helper; the default executor is always a mock. Namespace, OpenVPN, SOCKS5, Kill Switch and node-scanning services provide deterministic resources, bounded readiness checks, verified PID handling and rollback-safe cleanup. Fast scans normalize DNS/TCP/UDP outcomes without exposing exception text. Full scans use reserved Namespace slots, a firewall-only Kill Switch, fixed HTTPS exit/speed probes and fail-closed cleanup; simulated outcomes are persisted but never change real node availability. The connection-scoped firewall prefers nftables with an iptables fallback, permits only the configured OpenVPN remote before `tun0`, and binds proxy startup to the protected port. SOCKS5 credentials use a private Fernet key and connection-bound authenticated ciphertext; plaintext is excluded from helper arguments and retained only in short-lived private specifications plus the root-owned 3proxy runtime config.

Connection health and switch orchestration default to a deterministic simulated driver. Candidate selection excludes unavailable/current/blocked nodes and applies cached latency, speed and network-type policy before making changes. Real switching stops SOCKS5, verifies and preserves the Kill Switch, stops the old OpenVPN process, applies the new endpoint-specific rules, validates the new exit/IP class/service checks, and only then resumes SOCKS5. Database-backed events cap automatic attempts at five per connection per rolling hour. The background monitor and the real switch driver have separate opt-in settings; the latter also requires every lower-level real network gate.

Production maintenance uses the repository scripts for locked install, upgrade, repair, diagnosis, offline password reset and uninstall workflows. Configuration is passed to the service account through a validated clean environment; Alembic performs every schema change. Non-interactive passwords use a private one-use file rather than argv, while upgrades retain a root-only application/configuration/database backup and automatically restore it if migration or health verification fails.

Application startup is fail-closed for interrupted connections. Mock deployments mark unverified connections stopped and SOCKS endpoints inactive without calling the executor. Real connection deployments first invoke the fixed connection-purge helper operation for each affected connection and abort startup if any managed runtime resource cannot be cleaned. SQLite engines explicitly enable foreign-key enforcement so connection deletion cascades to dependent endpoint and check records.

Stage-fourteen acceptance covers two isolated simulated exits end to end, startup recovery, release reproducibility, credential exclusion, manifest integrity and archive traversal rejection. `make release-check` combines the complete test/build suite with Alembic round-trip validation and release verification while refusing enabled real-network gates.
